# Evaluation Protocol

This document explains how to evaluate DT/FDS without flattening all claims into one epistemic level.

## 1. Do Not Evaluate the Archive as a Single Claim

DT-Archive is a claim-space archive.
FDS-0 is the formal core.
FDS-P0 is the physical bridge registry.
FDS-A1 is an operational AI application.

## 2. Identify the Claim Layer

Before criticizing a claim, identify its layer:

| Layer | Description | Failure Mode |
|---|---|---|
| L0 | formal definitions and conditional theorems | mathematical counterexample |
| L1a | bridge-free finite-system consequences | counterexample under stated hypotheses |
| L1b | physical bridge claims | experimental or thermodynamic violation |
| L2 | domain applications | failed mapping or failed operational test |
| L3 | high-risk interpretive extensions | local demotion/quarantine |

## 3. Required Critique Form

A valid critique should specify:

1. Claim ID.
2. Target layer.
3. Stated dependency being challenged.
4. Counterexample or empirical condition.
5. Consequence: revise, demote, quarantine, or abandon.

## 4. Invalid Critique Forms

The following are insufficient:

- "This is not mainstream."
- "This resembles prior work."
- "Future engineering may solve it."
- "This is too broad."
- "This claim is speculative, therefore the formal core fails."

## 5. Valid Critique Examples

**Example:**

If a system satisfies all stated Landauer conditions and reliably performs logically irreversible erasure below the thermodynamic floor, then the Landauer bridge fails.

This does not automatically falsify the formal FDS core.
It demotes bridge-dependent thermodynamic claims.

## 6. Core Rule

Failure propagates downstream, not automatically upstream.
