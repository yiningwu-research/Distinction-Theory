# DT/FDS Release Protocol

Every major DT/FDS release should include:

1. A Git tag.
2. A GitHub release.
3. A PDF artifact.
4. Source files.
5. Claim ledger update.
6. Timestamp table update.
7. DOI or DOI-pending status.
8. Website link.
9. Failure-condition update if relevant.
10. Checksums for major files.

## Tag Format

Recommended tag format:
- `dt-archive-v1.0`
- `fds-core-v1.0`
- `fds-p0-v1.0`
- `fds-t1-v1.1`
- `fds-l1-submitted-v1.0`
- `fds-x1-pre-euclid-v1.0`
- `claim-ledger-v1.0`
- `priority-infrastructure-v1.0`

## Release Title Format

`FDS-T1 v1.1 — Finite Observer Distinguishability Budget Revision`

## Release Checklist

- [ ] Source files tagged and committed.
- [ ] PDF artifact attached.
- [ ] CLAIM_LEDGER.md updated with new or revised claims.
- [ ] TIMESTAMPS.md updated.
- [ ] VERSION_HISTORY.md updated.
- [ ] DOI created or updated on Zenodo.
- [ ] Website updated (papers, timestamps, ledger pages).
- [ ] Failure conditions reviewed.
- [ ] Checksums generated.
